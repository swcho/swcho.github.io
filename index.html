RabbitMQ 개념

By Sungwoo

Published 2016-02-15

Consumer, Producer

Producer는 message를 생성하고, Consumer는 message를 소비 한다.
AMQP의 명령을 보내는 가상의 연결인 Channel은 하나의 AMQP 연결(TCP)에서 여러개 설정할 수 있다.

Queue

Q를 기준으로 consumer는 메시지를 소비하며, “basic.consume”, “basic.get”으로 메시지를 가져온다.
하나의 Q에 여러개의 consumer가 구독하고 있다면, round robin 방식으로 메시지가 분배 된다.
consumer는 명시적으로 basic.ack를 주거나 Q 구독시, auto_ack 파라미터를 설정하여 자동으로 ack를 줄 수 있다.
RabbitMQ는 ack를 받으면 전달한 메시지를 삭제한다.
만일 consumer가 ack를 전달하기 전에 연결을 해제 하면, 메시지가 전달되지 않은 것으로 간주하고 다음 consumer에게 전달한다.
따라서 consumer 애플리케이션이 특정 메시지 처리 중 crash가 발생하면, 해당 메시지는 다른 consumer에게 전달되었다고 생각해도 무방하다.
하지만, consumer 애플리케이션이 버그로 인해 ack 전달을 하지 못했다면, 해당 consumer에게 더 이상 메시지를 전송하지 않는다.
만일, 명시적으로 ack를 하지 않고 reject를 하고 싶다면, 연결을 해제하거나 RabbitMQ 2.0 이상 부터 지원하는 basic.reject 명령을 사용한다.
RabbitMQ가 reject를 받으면 해당 메시지를 다음 consumer에게 전달한다.
Q는 producer, consumer 모두 queue.declare 명령을 통해 정의할 수 있다.
Consumer가 다른 Q를 구독 중일 때에는 Q를 정의할 수 없다.
Q는 다음과 같은 속성이 있다.

  • exclusive: 하나의 consumer만 접근
  • auto-delete: 마지막 consumer가 구독을 해지하면 삭제됨
    Q가 존재하는지 확인하려면, queue.declare 명령의 passive 옵션을 준다. 존재할 경우 성공을 그렇지 않을 경우 Q를 생성하지 않고 error를 반환한다.

Exchanges and bindings

$channel->basic_publish([message], ‘’, [exchange name]);
$channel->queue_bind([queue name], ‘’, [topic])
basic_publish(exchange, routing_key, body, properties=None, mandatory=False, immediate=False)
queue_bind(queue, exchange, routing_key=None, arguments=None)

  • Direct Exchange
    • $channel->basic_publish($msg, ‘’, ‘queue-name’);
  • Fanout Exchange

  • Topic Exchange

    • $channel->basic_publish($msg, ‘logs-exchange’, ‘error.msg-inbox’);
    • $channel->queue_bind(‘msg-inbox-errors’, ‘logs-exchange’, ‘error.msg-inbox’);
      error 메시지만 수신
    • $channel->queue_bind(‘msg-inbox-logs’, ‘logs-exchange’, ‘*.msg-inbox’);
      msg-inbox 모듈의 모든 메시지 수신
    • $channel->queue_bind(‘all-logs’, ‘logs-exchange’, ‘#’);
      모든 로그 수신

  • Exchange, queue, bindings

  • Queue는 binding rule에 의해 exchange에 연결한다
  • 메시지는 exchange로 전달한다.
  • direct, fanout, topic 의 3가지 exchange type이 존재한다.
  • 메시지 routing key와 exchange type에 따라 브로커는 메시지를 Q에 전달한다.

하나의 rabbitmq 내에서 가상의 호스트를 구성함으로서 독립적인 exchange, binding, queue를 가질 수 있으며, 이에 따라 권한 설정을 달리 가져갈 수 있다.

일반적으로 q의 내용은 RabbitMQ가 재부팅하면 사라진다. 하지만, persistence 설정을 하면 유지할 수 있다.

Effective Monitoring

By Sungwoo

Published 2016-01-28

데이터 수집

  • White box

    • Log parser: 로그 파일에서 특정 패턴의 정보를 추출하는 방식
    • Log scanner: 로그 파일에서 특정 패턴의 개수를 측정
    • Interface readers: System/Device interface를 읽음 (ex, linux /proc)

  • Black box

    • Prober: 시스템에 요청을 보내서 응답을 확인하는 방식 (ex, HTTP 요청)
    • Sniffers: 네트워크 인터페이스 트래픽 분석

Coverage

  • Resources
    • Hardware: CPU, I/O, Memory
    • Network: Packet loss, Latency, Bandwidth use
  • Software
    • OS: Process, Swapping, Free space
    • Middleware: Request, Cache size
    • Application: Availability, Registered users
  • Experience
    • Web Analytics: Split testing

Metrix

통계치

  • n
    인터벌 사이의 입력 개수
  • sum
    모든 값의 합
  • avg = sum/n
  • p0-p100
    입력 값의 100 분위 수
  • deviation
    입력한 값의 분산

Metrix 특성

단위

  • Amount: packet size, price, free disk space
  • Time Delay: WEB Request/Response Time, ICMP round trip time, time spent by a users
  • Amount per Time: bit rate, IOPS (Input/Ouput Operations Per Second)

HTTP 서버 예

  • Amount: 요청 크기,
  • Time Delay: 응답 시
  • Amount per Time: 시간당 요청 수

데이터 수집 모드

  • Active: ICMP ping / HTTP GET
  • Passive: Sniffing

데이터 소스

  • Internal: log date, device statistics
  • External: active 데이터 수집 모드 사용

Data Point 당 입력 수

  • Multi-N: 복수의 입력을 통해 만들어진 복수의 통계치, bytes 전송량, HTTP 요청 수
  • Single-N or 1-N: 단일 입력

양

  • Flow: 이벤트 성, Multi-N 같은 것; 패킷 전송량, 응답 시간
  • Throughput: 특정 시간 단위로 flow를 잰 것, bit rate, IOPS
  • Stock: 특정 시각에 쌓인 양, memory usage, free disk space, queue length, temperature, volume level
  • Availability: 특정 기대값에 미치는지 판단, ICMP ping 결과, 패킷 로스
